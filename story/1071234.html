如何使用 CSRF 和 JSON 类型混淆来提高您在 okcupid 上的受欢迎程度

2021-08-03 01:48:13

这是一篇关于安全漏洞（现已修复）的快速博客文章，它允许我让任何人喜欢或发送消息给 okcupid.com 上的个人资料，只需让他们点击我网站上的链接即可。在这样做时，我使用了一个最无聊的 Web 应用程序安全问题 (CSRF)，并结合了一个有点有趣的 JSON 类型混淆。证明它在一个同意帮助我进行安全测试并且绝对不是兔子的朋友身上起作用的证据：这个故事和许多人一样，开始于我打开开发工具并检查网站是否正在发送 CSRF 令牌以及需要身份验证的请求，例如发送消息从您的帐户到另一个用户。 CSRF 是一种攻击，攻击者通过该攻击向受害者发送链接，当在受害者的浏览器中访问该链接时，该链接会代表受害者在他们登录的站点上执行某些操作。例如，如果 Bob 登录了 Facebook，而 Facebook 在删除用户帐户的端点上没有 CSRF 缓解措施，那么 Alice 可以通过向他发送一个指向 https://facebook 的隐藏链接来诱使他删除他的 Facebook 帐户.com/delete（这不是删除您的 FB 帐户的实际 URL）。在这种情况下，我注意到 OkCupid 消息是通过 POST 请求发送到 https://www.okcupid.com/1/apitun/messages/send 的，带有 JSON 编码的正文，如下所示：显然，没有发送 CSRF 令牌请求。 CSRF 令牌是通过将不可猜测的秘密令牌包含为附加 HTTP 标头或 POST 参数来减轻这些攻击的常用方法。接收端点只有在验证令牌后才允许请求；这个想法是，虽然 Bob 的身份验证 cookie 在他点击 Alice 的恶意链接时由他的浏览器自动发送，但 Alice 无法在链接中包含有效的 CSRF 令牌，因此请求将失败。 Web 开发人员应该意识到 Web 最基本的安全策略是同源策略：在浏览器中运行的单一源（也称为方案 + 主机元组）的网站不应该能够访问不同源上的数据，除非另一个源通过跨源资源共享（CORS）等机制明确允许它。如果您的网站不发送 CORS 标头，您可能会认为 CSRF 是不可能的。问题在于某些类型的请求并不关心 CORS，并且对此的规则并不明显。例如：

如果您的站点尝试使用 XHR 获取跨源图像，默认情况下它会被阻止。如果您的站点通过 fetch API 将 POST 发送到另一个源，则默认情况下它会被阻止。如果您的站点通过 JavaScript 单击 HTML 表单元素上的提交按钮来发送 POST，则它可以工作。那么我们如何创建一个向 OkCupid 消息发送端点发送跨域 POST 请求的网页呢？这是尝试 #1：<html> <body> <form id="form" method="post" action="https://www.okcupid.com/1/apitun/messages/send"> <input style= 'display:none' name='foo' value='bar'> <input type="submit" value="Click me"> </form> <script> window.onload = () => {form.submit( )} </script> </body></html> 我在本地主机服务器上访问了这个，并且在表单自动提交后收到了一个非常有用的错误消息：有趣 - 它没有抱怨请求是由某个随机来源发起的，而是 POST 主体 (foo=bar) 是无效的 JSON！也许我们可以用一个奇怪的技巧来解决这个问题：

<html> <body> <form id="form" method="post" action="https://www.okcupid.com/1/apitun/messages/send"> <input style='display:none' name ='{"foo":"' value='bar"}'> <input type="submit" value="Click me"> </form> <script> window.onload = () => {form.submit ()} </script> </body></html> 我的理由是浏览器将 HTML 输入元素上的 name 和 value 属性转换为通过网络发送的 POST 正文中的 name=value ；所以如果 name 是 {"foo":" 并且 value 是 bar"}，这将变成 {"foo":"=bar"}，它将被 JSON 解析为看起来很奇怪的对象 {foo: '=bar'}没有问题。自从大流行之前就充满了希望，我在浏览器中加载了它并看到：......完全相同的错误。但这一次是因为我漂亮的 POST 正文已被 URL 编码为丑陋但安全的字符串 %7B%22foo%22%3A%22%3Dbar%22%7D。要是有办法告诉浏览器不要对这个表单体进行编码就好了！幸运的是，W3C 神灵以 enctype 属性的形式（双关语意为）给了我们这样的礼物。 <html> <body> <form id="form" method="post" enctype="text/plain" action="https://www.okcupid.com/1/apitun/messages/send"> <输入样式='display:none' name='{"foo":"' value='bar"}'> <input type="submit" value="Click me"> </form> <script> window.onload = ( ) => {form.submit()} </script> </body></html> 在 enctype="text/plain" 的帮助下，payload 最终以 { 的真实形式（再次双关语）发送"foo":"=bar"}: 把它们放在一起，下面的 HTML 会自动向假用户 ID 123 发送一条消息，说“我是一只兔子”。（找出某人的真实用户 ID 作为练习留给读者。）

<html> <body> <form id="form" method="post" action="https://www.okcupid.com/1/apitun/messages/send" enctype="text/plain"> <输入样式='display:none' name='{"foo":"' value='", "receiverid":"123", "body":"i am a rabbit", "source":"desktop_global", "service ":"other"}'> <input type="submit" value="点击我"> </form> <script> window.onload = () => {form.submit()} </script> </ body></html> 注意 POST body 变成了 {"foo":"=", "receiverid":"123", "body":"i am a rabbit","source":"desktop_global", "service “：“其他”}。因为浏览器在 name 和 value 属性之间插入了一个 =，所以它被设置为一个虚拟键 foo 的值，幸运的是 okcupid 忽略了它。有用！ … 或者，如果我完成了我的测试配置文件，那么我就可以向其他用户发送消息。懒得自己做，我把我的CSRF链接，填写了我自己的userID发给了一些朋友。瞧，我的 OkCupid 测试配置文件被一系列他们无意发送给我的消息所笼罩。除了让其他用户在不知情的情况下向您的 OkCupid 个人资料或您选择的其他人的个人资料发送消息外，我发现您可以使用本质上相同的漏洞让其他用户“喜欢”您的个人资料。显然，你可以滥用它来匹配任何你可以欺骗点击链接的人，或者你可以将链接垃圾邮件发送给一群人，以提高你的个人资料在 OkCupid 用来推荐人的任何神秘算法中的排名。我还想到，如果我将我的网站重定向到自动向我发送消息的 CSRF 链接，我可以看到登录到 okcupid.com 的网站访问者的 OkCupid 个人资料，这将成为一个强大的网络分析工具.显而易见的解决方法是 OkCupid 需要这些经过身份验证的端点的 CSRF 令牌，但我的攻击还依赖于这样一个事实，即这些端点乐于接受内容类型为 text/plain 的 POST，即使它们实际上期望的是 JSON。这让我很好奇其他站点是否也犯了同样的错误，所以我迅速抓取了一些 Alexa 顶级站点，寻找对包含 api 或 json 的端点的请求。

一旦我有了几百个端点的列表，我就向每个端点发送了一个内容类型为：text/plain 和 body {"foo":"bar"} 的 POST。这些端点中的 215 个中有 87 个没有出错，而且许多似乎返回 JSON 响应表示成功。当然，其中大部分可能不是经过身份验证的端点，其中一些可能需要接受非 JSON 文本，但这对我来说表明开发人员应该小心接受解析 JSON 的端点上的文本/纯文本输入。这已于 2021 年 4 月报告给 OkCupid。据他们的团队称，该问题已得到及时修复，从而阻止了该漏洞的利用。我要感谢他们的安全团队的赏金和分享这篇博文的许可。
