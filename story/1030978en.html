“changeme”是有效的base64

2020-10-25 03:08:59

Base64-encoding is ubiquitous in ourmodern world. Many programs communicate in base64-encoded messages, since thesehave nice properties: They consist of a very limited subset of ASCII characters,and can thus be displayed as text. E-Mails for example are commonly encoded inbase64 in transfer, which can be seen in the Content-Transfer-Encoding header.

Base64编码在我们的现代世界中无处不在。许多程序使用Base64编码的消息进行通信，因为这些消息有很好的属性：它们由非常有限的ASCII字符子集组成，因此可以显示为文本。例如，电子邮件在传输中通常以base64编码，这可以在Content-Transfer-Encoding报头中看到。

Where I volunteer we have a tradition of putting changeme as a placeholder formany template variables indicating that they should be overwritten later. Thus,if we later still see changeme anywhere we know that somewhere somebody musthave forgotten to specify a variable 1.

在我自愿的地方，我们有一个传统，将changeme作为任何模板变量的占位符，表明它们应该在以后被覆盖。因此，如果我们后来仍然在任何地方看到changeme，我们知道一定是有人忘记指定变量1了。

Sometimes these template variables should correspond to base64-encoded content,i.e. the base64-decoding of said variable should produce meaningful content. Butwhat happens if somebody forgets to define said variable, and it is substitutedwith changeme instead? I always thought that it would fail todecode the string since the probability that changeme is actually valid base64encoding must be very low. But lo and behold, this is not the case:

有时这些模板变量应该对应于base64编码的内容，即所述变量的base64解码应该产生有意义的内容。但是，如果有人忘记定义所述变量，而代之以changeme，会发生什么情况呢？我一直认为解码字符串会失败，因为changeme实际上是有效的base64编码的概率肯定非常低。但是你看，事实并非如此：

% echo changeme | base64 -d | hexdump -C00000000 72 16 a7 81 e9 9e |r.....|00000006

%ECHO CHANGEME|base64-d|十六进制转储-C00000000 72 16 A7 81 E9 9E|r.|00000006。

Mostly nonsense, but the second byte is interesting: It is an ASCII controlcharacter 2, meaning trips up all sorts of programs which expect to readprintable ASCII characters, such as JSON deserializers.

大部分都是胡说八道，但是第二个字节很有趣：它是一个ASCII控制字符2，这意味着所有期望可打印的ASCII字符的程序都会出错，比如JSON反序列化程序。

This bug actually occurred when I was migrating some applications betweendifferent Kubernetes clusters: In the old cluster the application ran withoutproblems, but I forgot to specify some variable in the new cluster, and it leadto Go complaining about non-ASCII characters in a JSON string.

这个错误实际上是在我在不同的Kubernetes集群之间迁移一些应用程序时发生的：在旧的集群中，应用程序运行没有问题，但是我忘记在新的集群中指定一些变量，这导致我抱怨JSON字符串中的非ASCII字符。

I guess the moral of the story is: If you use dummy values be sure that theycannot be misinterpreted.

我想这个故事的寓意是：如果你使用虚构的值，请确保它们不会被曲解。
