Apple M1 Microarchitecture Research
Apple M1微架构研究

This is an early attempt at microarchitecture documentation for the CPU in the Apple M1, inspired by and building on the amazing work of Andreas Abel, Andrei Frumusanu, @Veedrac, Travis Downs, Henry Wongand Agner Fog. This documentation is my best effort, but it is based on black-box reverse engineering, and there are definitely mistakes. No warranty of any kind (and not just as a legal technicality). To make it easier to verify the information and/or identify such errors, entries in the instruction tables link to the experiments and results (~35k tables of counter values).

这是在Apple M1中的CPU中的微型建筑文件的早期尝试，灵感来自Andreas Abel，Andrei Frumusanu，@Veedrac，Travis Downs，Henry Wongand Aganter Fog的惊人工作。本文档是我的最佳努力，但它基于黑匣子逆向工程，肯定会有错误。没有任何形式的保修（而不仅仅是作为法律技术性）。为了更容易验证信息和/或识别此类错误，指令表中的条目链接到实验和结果（〜35K计数器值）。

These are refered to as "units", to try to avoid confusion if Apple releases official documentation, as theyprobably refer to them as "ports" or "pipes", and order them differently. (If this just causes more confusion,I apologise.)

这些被称为＆＃34;单位＆＃34;如果Apple发布官方文件，则尝试避免混淆，因为他们可以将它们称为＆＃34;端口＆＃34;或者＆＃34;管道＆＃34;，它们不同地订购。 （如果这只是引起更多的混乱，我道歉。）

Integer units: 1: alu + flags + branch + adr + msr/mrs nzcv + mrs 2: alu + flags + branch + adr + msr/mrs nzcv 3: alu + flags + mov-from-simd/fp? 4: alu + mov-from-simd/fp? 5: alu + mul + div 6: alu + mul + madd + crc + bfm/extrLoad and store units (up to 128-bit loads and stores, including address generation with shifts up to LSL #3): 7: store + amx 8: load/store + amx 9: load 10: loadFP/SIMD units: 11: fp/simd 12: fp/simd 13: fp/simd + fcsel + to-gpr 14: fp/simd + fcsel + to-gpr + fcmp/e + fdiv + frecpe + frsqrte + fjcvtzs + ursqrte + urecpe + sha

整数单位：1：alu +标志+分支+ ADR + MSR / MRS NZCV + MRS 2：ALU + FLAGS + BRANCL + ADR + MRS / MRS NZCV 3：ALU + FLAGS + MOV-FROM-SIMD / FP？ 4：alu + mov-from-simd / fp？ 5：ALU + MUL + DEV 6：ALU + MUL + MADD + CRC + BFM / extload和商店单位（最多128位负载和商店，包括转移到LSL＃3的地址生成）：7：Store + AMX 8：负载/存储+ AMX 9：负载10：LoadFP / SIMD单元：11：FP / SIMD 12：FP / SIMD 13：FP / SIMD + FCSEL +至-GPR 14：FP / SIMD + FCSEL +至GPR + FCMP / E + FDIV + FRECPE + FRSQRTE + FJCVTZS + URSQRTE + URECPE + SHA

Certain instructions are able to issue as one uop if they appear consecutively in the instruction stream.

如果在指令流中连续出现，则某些指令能够作为一个UOP发出。

adds/subs/ands/cmp/tst + b.cc (complete fusion when fused instructions read no more than 4 registers per 6 instructions)

添加/潜艇/和/和CMP / TST + B.CC（完全融合当融合指令时每6条指令读取不超过4个寄存器）

pmull + eor (usually fused if operands match pattern "A, B, C ; A, A, D" or "A, B, C ; A, D, A")

PMULL + EOR（通常融合如果操作数匹配模式＆＃34; a，b，c; a，a，d＆＃34;或＆＃34; a，b，c; a，d，a＆＃34;）

amx + amx (excluding loads and stores - probably fuses to something like a STP)

AMX + AMX（不包括负载和商店 - 可能融合到STP的内容）

mov imm/movz/movn (handled by renamer at a max of 2 per 8 instructions, includes all tested "mov")

MOV IMM / MOVZ / MOVN（每8条指令的最大2个由Renamer处理，包括所有测试＆＃34; MOV＆＃34;）

MADD's output can be passed to its third operand (the addend) with 1c latency, but if it's chained with other instructions it has 3c latency.

MADD＆＃39; S输出可以通过1c延迟传递给第三个操作数（附加品），但如果它＆＃39;与其他指令链接，它有3c延迟。

Loads may be passed to the base address of other loads with 3c latency (which is nice for linked lists), but chaining with out ALU operations gives a latency of 4c. (Although the second destination register in an LDP always has 4c latency.)

负载可以传递给具有3C延迟的其他负载的基础地址（对于链接列表很好），但用alu操作链接给出了4c的延迟。 （尽管LDP中的第二个目的地寄存器始终具有4C延迟。）

Integer to SIMD/FP roundtrip latency can be as low as 7c (e.g. when chaining flags operations).

SIMD / FP往返延迟的整数可以低至7C（例如链接标志操作时）。

Firestorm can retire eight instructions per cycle, but can issue more uops (using implicit shifts or extends on ALU operations, as thus far other uops retire separately).

Firestorm每周期可以退出八个指令，但可以发出更多UOPS（使用隐式移位或在ALU操作上延伸，因此其他UOPS单独退休）。

These numbers mostly come from the M1 buffer size measuring tool.The M1 seems to use something along the lines of a validation buffer, rather than a conventional reorder buffer, whichcomplicates measurements a bit. So these may or may not be accurate.

这些数字主要来自M1缓冲区大小测量工具。M1似乎在验证缓冲区的行中使用某些东西，而不是传统的重新排序缓冲区，它符合测量值。所以这些可能或可能不准确。
