Resolvd(8) – daemon to handle nameserver configuration

With the following commit, Florian Obser ( florian@) imported resolvd(8), a daemon for handling nameserver configuration:

随着以下 犯罪， 进口Florian Obser（florian @） resolvd（8）， 用于处理名称服务器配置的守护程序：

CVSROOT: /cvs Module name: src Changes by: florian@cvs.openbsd.org 2021/02/24 11:10:41 Added files: sbin/resolvd : Makefile resolvd.8 resolvd.c Log message: Import resolvd(8), a daemon to rewrite resolv.conf. prodding deraadt

CVSROOT：/ cvs 模块名称：src 更改者：florian@cvs.openbsd.org 2021/02/24 11:10:41 添加的文件： sbin / resolvd：Makefile文件resolvd.8 resolvd.c 日志信息： 导入resolvd（8），这是一个重写resolv.conf的守护程序。 催促

resolvd handles the contents of /etc/resolv.conf, which contains details of the system's DNS nameservers, and is read by the resolver routines in the C library. resolvd checks whether unwind(8) is running and monitors the routing socket for proposals sent by dhclient(8), slaacd(8), or network devices which learn DNS information such as umb(4).

resolvd处理/etc/resolv.conf的内容，其中包含详细信息 系统的DNS名称服务器，由解析器例程读取 C库。 resolvd检查unwind（8）是否正在运行并监视路由 dhclient（8），slaacd（8）或网络设备发送的投标的套接字 可以学习DNS信息，例如umb（4）。

Do we really need a daemon for this? I agree that dnsmasq can be useful for certain things like you have various internal .dom1.lan .dom2.lan etc name servers and you want to route the request to the proper NSes but as I said we have dnsmasq, why to make something else now?!? Also in normal scenarios it is not even needed, why do we need another useless daemon running all the time for name resolution?

我们真的需要一个守护进程吗？我同意dnsmasq可用于某些事情，例如您具有各种内部.dom1.lan .dom2.lan等名称服务器，并且您希望将请求路由到适当的NSes，但是正如我所说的，我们有dnsmasq，为什么还要做其他事情现在？！？同样在正常情况下甚至不需要它，为什么我们需要一直运行另一个无用的守护程序来进行名称解析？
